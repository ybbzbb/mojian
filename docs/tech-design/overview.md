# Tech Design · Overview

> 本篇是 mojian 执行器的**项目级技术设计总纲**（理念定型阶段的设计草案，多数为 `planned` / `TODO`，未由代码证明，不作为已实现事实）。文件夹索引见 `docs/tech-design.md`（WIKI）。总纲 RFC 见 GitHub #1，三个 SOP 设计见 #5/#6/#7。

## 一句话架构

**mojian = SPEC（提示词资产）+ 确定性执行器（Rust）。**

生成链：

```
SPEC + 输入      → 过程产物
SPEC + 过程产物  → 结果产物
```

程序（状态机）只管两件事：**现在该跑哪个 SPEC、喂它哪些切片**。创意与品味归模型和人，程序不碰。

## 两个作用域：客户端（程序） vs 项目（运行环境）

核心分界：**程序是唯一的机器状态源；项目只是运行环境。**

```
mojian 客户端（程序 = 唯一的机器状态源，装一次，全局）
├── SPEC 权威副本（版本化，带 hash）
├── 中央 DB（一个库，按 project_id 分区）
│     项目登记 · 各项目状态 · 统计 · 配置 · artifact_ref · bible_version/void
├── 日志文件（按 project_id）：generation / decision / check（只增不改，非 DB）
└── 全局默认配置
        │  启动 / 打开项目
        ▼
项目（运行环境 = 一个目录，只有两类东西）
├── SSOT           创作内容（过程产物 + 结果产物），人直读直改；程序读它做输入、写它做产出
└── SPEC 部署缓存   可弃；启动时按 hash 比对，不一致直接覆盖（题材覆盖在客户端管理）
```

- **状态在逻辑上跟着项目**（每项目一份），**物理上存在客户端中央 DB**（按 `project_id` 分区）。
- 项目目录里**没有机器状态**——光拷项目目录到另一台机器拿不到进度/日志，需客户端导出/同步（集中管理的既定代价）。
- 项目内的 SPEC 是**纯可弃缓存**，永远以客户端为准；作者的题材风格覆盖不在项目里改，而由客户端按项目管理（选项 A）。

## 三层数据边界（名词 + 物理归属，全项目统一，不得混叫）

| 层 | 是什么 | 物理归属 |
|---|---|---|
| **SPEC** | 所有提示词 + 规则（每步「读什么/干什么/输出什么/自检闸门/禁止项/VOID 怎么走」） | 权威在**客户端**；项目里是可弃部署缓存 |
| **SSOT** | 人直读直改的创作内容：**过程产物**（抽取信息、风格说明、圣经…）+ **结果产物**（大纲、正文…） | **项目**目录 |
| **DB** | 机器变量：状态 · 统计 · 配置 · 引用（日志走文件）。**不是创作内容** | **客户端**中央 SQLite（按 project_id 分区）+ 日志文件 |

## 启动执行流

```
1. 打开项目 → 比对项目 SPEC 缓存 hash vs 客户端权威 → 不一致直接覆盖重部署
2. 从客户端中央 DB 取该项目状态
3. 执行循环（token 只花在 SDK.跑 一步，其余是免费的确定性 Rust）：
   loop {
     state  = 客户端DB.读该项目状态()      // phase + 内部状态
     action = 状态机.下一步(state)          // 纯函数，零 token —— 压 AP-001
     if action.要人 { 停，等人 decide }     // 到关卡了
     bundle = 装配器.切片(action, state)    // 算出最小 SPEC+SSOT 切片 —— 压 AP-002
     result = SDK.跑(bundle)               // claude 无头子进程，产出写回项目 SSOT
     checks = 检查器.客观校验(result)        // 字数/对话占比/结构，零 token —— 压 AP-003
     客户端.落库(action, result, checks)    // 状态/统计写 DB，日志追加文件
   }
```

细节：状态机见 `domain-model.md`；部署、切片、SDK 调用、检查器见 `engine.md`。

## 升级模型（集中带来的红利）

机器状态全集中在客户端，三个升级动作都只动一处、不用逐项目迁移：

- **SPEC 升级**：客户端权威更新 → 项目启动时 hash 不一致自动覆盖重部署。
- **DB schema 升级**：只在客户端中央库跑一次迁移。
- **默认配置升级**：默认值随程序；项目只存覆盖项，没覆盖的自动吃新默认。
