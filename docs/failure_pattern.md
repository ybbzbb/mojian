# Failure Patterns

> 本项目尚未产生自己的实现级 bug。以下 Anti-Patterns 是从 **mojian 前身（一套私有 markdown 多 Agent 创作协议，实测产出 94 章 / 32 万字）** 的真实教训中提炼的结构性失败模式——它们正是 mojian 三个 SOP 的设计动因，必须在执行器与提示词实现中主动规避。`Recurring Bugs` 待项目自身实现后沉淀。

## Anti-Patterns

### AP-001 状态机写成散文，靠 LLM「自律」转移

- 现象：状态转移规则写成自然语言，靠 LLM 阅读规则后自行遵守；既烧 token 又不可靠。
- 根因：把确定性调度逻辑交给了概率模型。
- 正确做法：状态机、调度、游标、版本转移必须是 Rust 确定性代码，零 token。提示词只负责生成/分析，不负责流程控制。

### AP-002 流程开销吞掉大部分 token（逐角色重复装载）

- 现象：每章 ~70% 的读取量是协议文档和 agent 合同的重复装载；多 agent 逐章冷启动放大成本。
- 根因：每个角色都把所有文件重新读一遍，无上下文最小集概念。
- 正确做法：由程序计算每步的最小读取文件集（圣经切片 · style · 前情 · 失败模式）；批次化调度，把冷启动从「每章 3-4 次」降到「每批 1 次」。

### AP-003 验证机器瞄错靶子（用一致性验证冒充品味验证）

- 现象：94 章 118 次 LLM APPROVE 仅换来 19 次 void——一致性风险早被圣经阶段压住，而真正的产品级失败（正文过于文学化、不抓人）被整条验证链系统性放行，首章重写至 v8 后项目停摆。
- 根因：把「客观一致性」和「主观品味」混为一谈，用高频 LLM 验证做了低价值的事，真正需要人判断的品味却无人把关。
- 正确做法：**一致性交给代码**（字数/对话占比/结构/时间线，程序检查，零 token）；**品味交给人**，且**先校准再量产**（每卷动笔前做品味校准盲选，人工关卡专注「爽不爽」）。

### AP-004 品味偏差被逐章验证放行，酿成大规模返工

- 现象：风格偏差被逐章 LLM 验证放行 90+ 章，最终整体返工——旧协议最贵的事故。
- 根因：缺少「先校准品味再批量生产」的关卡；逐章验证给了「已把关」的假象。
- 正确做法：SOP③ 每卷开工前做品味校准（2-3 风格变体人工盲选，结论固化进 style）；未校准且未显式跳过则程序拒绝进入批次循环。

### AP-005 内容违反圣经时改章节迁就，而非修正基因

- 现象：写作中发现内容与圣经冲突，倾向于就地改章节让它「圆过去」。
- 根因：圣经是最贵的错误（向下传播到几百章），但改章节比改圣经看起来省事。
- 正确做法：走 VOID 机制——作废相关章节 → 回 SOP② 修圣经 → 重走受影响的验证与确认 → 重写。圣经版本化、void 原因与影响范围入库、受影响章节自动标记。
